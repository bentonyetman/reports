TL;DR for leadership

CDS adoption is broad but shallow. Four out of five active frontend repos pull in a CDS package - but only about a quarter of UI files actually import from CDS, and more than a quarter use raw @mui/* components without the Capital theme wrapper. Without intervention, the gap will widen: we already see two parallel libraries forming and a shadcn/ui beachhead in six repos.

Three things to fix first

1. Close the two biggest adoption holes (bifrost, product-roadmap-portal) - they alone represent ~1,050 unthemed UI files. 2. Resolve the og-ui-components-library parallel-library situation before it grows further. 3. Stop the shadcn/ui pattern spreading - it's in six repos today.

What we found

1. Adoption is broad but shallow

CDS is the de-facto choice - 82% of active frontend repos pull in at least one CDS package - but day-to-day, only 23% of UI files actually import from CDS, and 26% import raw @mui/* without wrapping in the Capital theme. The typical pattern is "CDS for the theme and nav bar, then roll my own" rather than "CDS-first."

2. A small number of repos contain most of the gap

Two repos alone account for over a thousand UI files with effectively zero CDS adoption:

4. shadcn/ui has quietly landed in six repos

Six repos have the tell-tale components/ui/{button,card,dialog}.tsx shadcn-starter pattern

migration-orchestrator-app, product-roadmap-portal, spc-poc, visibility-portal, gab-311, gab-boilerplate

shadcn is a trendy open-source component starter that teams install in minutes - but it ships Tailwind and a set of primitives that completely bypass Capital. Left unchecked, this becomes a parallel design language inside our own codebase.

5. A parallel library is forming under the "og-ui-components-library" name

Despite the name, og-ui-components-library is an ERP-team monorepo, not a design system

Very active (32 commits / 30 merged PRs in the last 30 days) · No CODEOWNERS, no required reviews.

The repo's @opengov/account-code-entry package has already displaced CDS's @opengov/components-account-number-input across the ERP suite (erp-fin-ap, erp-fin-gl, pro-app, pro-purchase-order-api). ACE is genuinely better - but because it sits outside CDS governance, every future ERP-built component faces the same fork-or-upstream decision with no forum to resolve it.

Newer additions like AppEmptyState (overlaps CDS Result) and CapitalProgress (wraps MUI progress) are early warning signs: general-purpose primitives are starting to be re-minted outside CDS.

Recommendations

Ranked by likely impact. The first four are the ones I'd bring to a design-system steering review.

For the platform / CDS team

  1. Close the two biggest adoption gaps first
    bifrost (695 unthemed UI files) and product-roadmap-portal (359 files, shadcn) dwarf every other gap. One focused engagement per repo would move the org-wide CDS-import rate several points.
  2. Publish a stance on shadcn/ui
    Either formally bless it as a prototyping-only pattern with a written deprecation path, or replace it with CDS equivalents before more teams copy the pattern from the six repos that have it today.
  3. Sunset or promote - don't leave CDS stubs competing with ERP's parallel
    CDS's components-account-number-input has a single external consumer; the ERP team's ACE is used in four production apps. Either promote ACE into CDS or formally retire the CDS stub and point consumers at ACE.
  4. Publish a "which component do I use?" decision tree
    The fact that Button is being re-implemented in 12 repos suggests that teams can't quickly find the right CDS / MUI answer. A one-page decision tree (with paths to a CDS component, a themed MUI component, or "ask in #design-system") would eliminate most of the drift we measured.

For og-ui-components-library

  1. Rename the repo to reflect its ERP scope
    og-ui-components-library invites drift from every team. Something like erp-components or erp-shared makes the boundary honest and tells other teams not to shop there for generic UI.
  2. Add CODEOWNERS and required reviews
    A library shipped into four production apps with zero review gating is a governance hole. This is a 30-minute fix.
  3. Promote the strictly-better pieces into CDS
    ACE and the purchase-order / vendor-search components are high-quality and solve real org-wide problems. Formalizing ownership prevents other suites from doing the same parallel work.
  4. Stop general-purpose additions from drifting in
    AppEmptyState and CapitalProgress are both generic enough that they should either use existing CDS / MUI primitives or be proposed upstream to component-library. Set a rule and document it in the repo's README.

For frontend enablement broadly

  1. Wire code-cds-frontend checks into CI for the top adopter repos
    New drift should be caught at PR time, not in a scorecard six months later. The existing agent spec already encodes the rules.
  2. Re-run this scorecard quarterly
    The scan is fully repeatable (scripted, ~5 minutes end-to-end). Tracking the 23% "UI files using CDS" number over time is the single best signal of whether the design-system strategy is landing.
