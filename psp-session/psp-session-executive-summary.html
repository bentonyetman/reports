PSP Session Adoption

Executive summary · 2026-05-04 19:10 · 14 PSP-mountable repos (23 repos audited; 9 have no mount entry)

Headline

Remotes get the boundary right. 12 of 14 PSP-mountable repos (86%) consume the auth0Config PSP supplies. The contract is being honored at the mount call.

Each remote also creates its own <Auth0Provider>, ungated. Across the corpus, 1 of 26 Auth0Provider instantiations are wrapped in a runtime check (isStandalone / !options.auth0Config / equivalent). 12 of 14 repos instantiate one unconditionally. Runtime verification on staging shows these parallel Providers reuse the shell's clientId, so they share the same @@auth0spajs@@ localStorage cache - the runtime impact is softer than the static finding suggests (separate React state and refresh timers, but one shared token cache). The static fix - add the gate - is still correct hygiene; urgency is lower than "users have two different tokens".

2 repos drift harder than that - either bypassing PSP auth entirely or using a parallel mount-contract package. Worth handling separately.

Runtime turned up something the static audit missed. On the bare PSP shell at staging, two distinct Auth0 client IDs are already authenticated before any remote is navigated to: the shell's own client (audience opengov) and a second application (audience default) whose origin is unidentified. This is a separate Auth0 application, not the parallel-Provider pattern. Worth investigating before the gate work, since it represents a real cross-application session.

Contract governance is the longer-term concern. Only 2 of 14 (14%) import @opengov/psp-contract for types. 11 repos redefine the types locally; 1 import from a parallel package. The canonical contract has no single source of truth today.

Runtime verification (staging)

Static analysis can't see whether a remote initializes its Auth0Provider but then short-circuits at runtime, so the findings above were verified against a live signed-in session at psp.ogstaging.us.

ObservationImplication
Bare PSP shell (Command Center, no remote navigated to) has 2 distinct Auth0 client IDs in localStorage: vLbqnFiT... (audience opengov) and bC1CU3RK... (audience default). Each has its own auth0.<clientId>.is.authenticated cookie. Two separate Auth0 applications are signing in the same user. The second is not the parallel-Provider pattern; it's a cross-application session. Origin is unidentified - worth tracking down.
Mounting Vendor Management, Utility Billing, and Procurement & Contract Management each: did not add a new client ID to localStorage. The remotes' parallel <Auth0Provider> instances reuse the shell's clientId. Despite the static finding that every remote creates an ungated Provider, those Providers share the shell's @@auth0spajs@@ cache via the same clientId. Runtime impact is softer than "two diverging token caches"; closer to "two React contexts reading from one cache". Race conditions and split isAuthenticated state still possible, but no token-cache split.
Procurement remote made an /authorize call to login.ogstaging.us with client_id=vLbqnFiT... (the shell's clientId), audience=opengov. Confirms the remote's Auth0Provider acts on the shell's identity, not a separate one.

What this changes. The parallel-<Auth0Provider> pattern is real in code, but at runtime it's not producing two diverging tokens - it's producing two React contexts that share one token cache. The static fix (gate the Provider) is still right, but the production-safety urgency is lower than the static report alone implied. The genuinely unexpected runtime finding is the second Auth0 application (bC1CU3RK / audience default), which the static audit could not have caught.

Recommendations

  1. Investigate the second Auth0 application first. Runtime verification on staging found a second client ID (bC1CU3RK..., audience default) authenticated on the bare PSP shell, separate from the shell's own vLbqnFiT... / audience opengov client. That's not the parallel-Provider pattern this audit primarily covers - it's a separate Auth0 application. Find the OpenGov product registering it (Auth0 admin console, client_id=bC1CU3RK), decide whether it consolidates into the shell's app or remains intentionally separate. If users see session symptoms today, this is a more plausible cause than the parallel-Provider pattern.
  2. Handle 2 hard-drift repos separately. These bypass PSP auth in ways that are not just a missing gate (e.g., parallel mount contract, browser-side token storage, cookie session middleware). Each needs a targeted review.
  3. Add the standalone-mode gate to 15 repos (lower urgency). Every PSP-mountable remote needs its own Auth0 setup for standalone use - that's not the issue. The issue is that none of them gate it, so the standalone Auth0Provider also fires when PSP mounts them. Runtime data shows these parallel Providers share the shell's clientId and localStorage cache, so the production-safety risk is latent rather than active. Still worth fixing - one conditional per repo: {!options?.auth0Config && <Auth0Provider...>} (or equivalent). Publish a reference example in @opengov/psp-contract so teams can copy it.
  4. Pick one canonical contract package. 1 repos import PSP types from @opengov/app-core/types/psp instead of @opengov/psp-contract. Either deprecate the alternate or migrate the canonical to app-core - having both is the worst outcome.
  5. Migrate 11 repos with locally-defined types onto @opengov/psp-contract. Behavior is correct today but the contract has no enforced source of truth. When PSP adds a new field (e.g., a new Auth0Config property), nothing surfaces it to the remotes - they all silently start ignoring it.
  6. Add a CI lint rule. Once the reference gate pattern exists, an ESLint rule can fail PRs that introduce <Auth0Provider> in a file outside an isStandalone/!options.auth0Config conditional. Catches the next instance before it ships.
